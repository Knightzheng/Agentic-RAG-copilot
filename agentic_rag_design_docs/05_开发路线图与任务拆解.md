# 《Agentic RAG 开发路线图与任务拆解》

**文档版本**：V1.0  
**文档定位**：开发计划书 / 里程碑拆解 / Codex 协作执行清单

---

## 1. 总体实施策略

本项目不建议“从前端到 MCP 一次性全做完”，而应遵循：

**先主链路，后增强；先可运行，后高级化。**

推荐推进顺序：

1. 基础工程骨架
2. 文档上传与知识库
3. RAG 主链路
4. 引用与追踪
5. LangGraph 编排
6. 工具与审批
7. 记忆与 MCP
8. 评测与运维增强

---

## 2. 里程碑划分

## 4. 你与 Codex 的协作模式建议

### 4.1 你负责什么
- 决策整体方向
- 审核模块边界
- 监督实现是否偏离设计
- 学习关键模块
- 准备面试表达

### 4.2 Codex 负责什么
- 建目录结构
- 建模型与 migration
- 实现标准接口
- 实现模块代码
- 生成基础测试代码
- 补充工程化细节

### 4.3 协作原则
- 每次只让 Codex 完成一个清晰模块
- 输入前附上对应设计文档
- 明确要求：
  - 保持代码风格统一
  - 不随意越层耦合
  - 每个模块都写用途注释
  - 完成后给出变更说明

---

## 5. 推荐任务拆分粒度

### 好的粒度
- “实现 documents 与 document_versions 的 ORM 和 migration”
- “实现 chunking 服务和对应单元测试”
- “实现 retrieval service，不接前端”
- “实现 chat SSE 事件流”

### 不好的粒度
- “把整个 Agentic RAG 全部写完”
- “把前后端所有页面都实现”
- “自己决定项目架构并重写现有代码”

---

## 6. 风险点清单

### 6.1 技术风险
- 文档解析效果不稳定
- rerank 成本和延迟较高
- 向量检索与过滤组合的性能问题
- LangGraph 状态设计过于膨胀
- MCP 工具接入复杂度高

### 6.2 工程风险
- 模块边界不清，导致后续返工
- 先做高级能力，主链路反而不稳
- 前后端字段命名不统一
- 日志字段设计不完整，Trace 难补

### 6.3 项目管理风险
- 一次给 Codex 太大任务
- 缺少阶段验收标准
- 缺少可固定演示的 demo 数据

---

## 7. 每阶段验收模板

每完成一个阶段，建议按以下模板验收：

### 功能是否跑通
- 是否能从输入走到输出
- 是否能在前端触发与展示

### 数据是否落库
- 是否写入正确表
- 状态是否更新正确
- 是否能追溯来源

### 日志是否完整
- 是否有 run
- 是否有 step
- 是否有错误记录
- 是否有可视化 trace

### 可讲解性是否成立
- 能否讲清楚设计目的
- 能否展示输入、处理、输出全过程

- 完成 Chat 页面基础版
- 展示问答 + 引用

- 修复主链路问题

## 9. 结论

这份路线图的核心目的是：

- 避免项目一开始就失控
- 让你和 Codex 有明确任务边界
- 让整个项目始终沿着“可运行 → 可解释 → 可扩展”的路线推进
