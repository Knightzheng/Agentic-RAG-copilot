## 3. 部署拓扑建议

推荐最小部署拓扑：

```text
frontend
api-server
worker
postgres + pgvector
redis(optional)
object storage(optional)
```

### 3.1 Frontend
- React 静态资源
- 可部署在 Vercel、Nginx、静态服务器

### 3.2 API Server
- FastAPI
- 处理同步请求
- 负责 chat、threads、documents、memory、mcp、approval 等接口

### 3.3 Worker
负责异步任务：
- parse
- embedding
- eval
- memory summarize
- mcp sync

## 6. 安全设计

### 6.1 身份与权限
角色：
- admin
- editor
- viewer

### 6.2 数据隔离
所有以下表都必须强制带 `workspace_id` 边界：
- documents
- threads
- agent_runs
- memories
- mcp_servers

### 6.3 工具安全
所有工具需要：
- 参数 schema 校验
- 风险等级
- 审批规则
- 调用日志
- 调用超时控制

### 6.4 Prompt 安全
- 系统 prompt 不允许前端直接覆盖
- prompt 模板必须版本化
- 工具描述不可由普通用户动态修改

### 6.5 文档安全
- 文件类型白名单
- 文件大小限制
- 文件 hash 去重
- 防止恶意上传
- 敏感文档支持受限可见

### 6.6 MCP 安全
- server 白名单
- endpoint 审核
- tool 级 enable / disable
- 高风险 MCP tool 审批
- 认证配置加密保存

## 8. 告警建议

### 8.1 高优先级
- API 大面积 5xx
- 数据库不可用
- 向量检索异常
- embedding 服务持续失败
- MCP server 全部失联

### 8.2 中优先级
- 文档解析失败率升高
- rerank 超时率升高
- approval 堆积过多
- 队列 backlog 增长

---

## 9. 评测体系

### 9.1 评测目标
评测不是为了“做一个分数”，而是回答：
1. 检索是否正确召回了证据
2. 回答是否真正基于证据
3. Agent 是否能稳定完成任务
4. 系统是否具备稳定演示与迭代能力

### 9.2 离线评测
- 检索：Recall@K / MRR / nDCG / Hit@K
- 生成：Faithfulness / Relevance / Completeness / Citation Precision
- Agent：Tool success rate / Retry rate / Completion rate

### 9.3 在线评测
从真实运行数据中抽取：
- 低评分 run
- 无引用 run
- 证据不足 run
- 多次重试 run

进入人工 review 或 LLM-as-judge 流程。

## 11. 运维策略

### 11.1 数据备份
- 数据库定时备份
- 对象存储原文备份
- 配置与 prompt 版本备份

### 11.2 数据清理
- 冷日志归档
- 失败 job 清理
- 长期无用临时文件清理

### 11.3 模型变更策略
- embedding 模型升级走版本化
- rerank 模型升级做 A/B 对比
- prompt 升级记录版本号

---

## 12. 结论

部署、安全、评测、运维决定了这个项目是“一次性 Demo”还是“可维护、可展示、可扩展的工程项目”。

本项目必须做到：
- 主链路可部署
- 高风险动作可控
- 检索与回答可追踪
- 运行质量可评测
- 演示环境可重复
