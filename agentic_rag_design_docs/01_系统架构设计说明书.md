## 1. 项目概述

Atlas Agentic RAG Copilot 是一个面向“知识问答 + 工具调用 + 长短期记忆 + MCP 扩展 + 前端可演示”的 Agentic RAG 系统。

本项目的目标不是做一个“上传 PDF 然后问答”的基础 Demo，而是做一套可以支撑：

- 项目演示
- 求职面试讲解
- 后续持续扩展
- 前后端真实运行

的中高质量 AI 应用。

系统以 **LangGraph** 作为 Agent 编排底座，以 **LangChain** 作为 RAG 主链路与工具生态入口，以 **PostgreSQL + pgvector** 作为统一数据与向量存储底座，以 **React + TypeScript** 作为前端交互层，最终形成一个具有企业应用雏形的可扩展 Agentic RAG 产品。

---

## 2. 项目目标

### 2.1 总体目标

构建一个具备如下特征的 Agentic RAG 项目：

1. 架构清晰、模块边界明确
2. 技术路线先进，但复杂度可控
3. 前后端完整，可实际操作
4. 能支持面试中的系统设计讲解
5. 能支持后续由 Codex 或人工持续扩展

### 2.2 具体目标

系统需要具备以下能力：

- 文档上传、解析、切分、向量化、索引构建
- 多路检索、重排序、证据筛选、引用回答
- LangGraph 图编排
- ReAct 风格工具执行
- MCP 工具接入
- 3~4 层记忆体系
- Human-in-the-loop 审批与恢复
- 前端步骤可视化与引用可视化
- 运行日志、反馈与评测闭环

### 2.3 非目标

第一版不追求：

- 复杂且失控的多 Agent 社会模拟
- 超大规模多租户 SaaS 级基础设施
- 重度行业定制能力
- 全量自动化办公代理

## 3. 设计原则

### 3.1 可控优先于炫技

Agent 系统的重点不是“让模型自由发挥”，而是让系统：

- 可追踪
- 可中断
- 可恢复
- 可审计
- 可降级

因此，本项目采用 **Supervisor + 子图 + 条件路由** 的显式图结构，而不是单个黑盒 Agent。

### 3.2 检索质量优先于“接上向量库”

RAG 的真正质量差异，主要来自：

- 文档解析质量
- Chunk 结构设计
- 上下文化检索
- 混合检索
- 重排序
- 证据充足性判断

因此，本项目不会停留在“向量搜索 + LLM”层面，而会设计完整检索链路。

### 3.3 分层记忆优先于简单拼接历史消息

系统记忆不等于简单 chat history。  
本项目会区分：

- 工作记忆
- 情节记忆
- 语义记忆
- 程序性记忆

### 3.4 前端真实可操作优先于纯命令行展示

项目必须有：

- 聊天页
- 文档管理页
- 运行追踪页
- 记忆管理页
- MCP 管理页

才能真正体现“产品型 AI 项目”的价值。

## 5. 总体系统分层

系统采用六层结构：

1. 展示层（Frontend）
2. 接入层（FastAPI API Gateway）
3. 编排层（LangGraph Supervisor）
4. 能力层（RAG / Tool / MCP / Memory）
5. 存储层（PostgreSQL + pgvector）
6. 观测层（Logs / Trace / Feedback / Eval）

逻辑结构如下：

```text
Frontend
  ↓
FastAPI API Gateway
  ↓
LangGraph Supervisor Graph
  ├─ RAG Subgraph
  ├─ Tool Subgraph
  └─ Hybrid Subgraph
  ↓
Knowledge Base / Tool Layer / Memory Layer / MCP Layer
  ↓
PostgreSQL + pgvector + Logs + Eval
```

## 6. Agent 编排总体方案

### 6.1 主图：Supervisor Graph

主图负责：

- 读取线程上下文
- 分类请求
- 召回记忆
- 路由子图
- 汇总结果
- 写入记忆与日志

### 6.2 三个子图

#### A. Knowledge RAG Subgraph
负责知识检索问答。

#### B. Tool Execution Subgraph
负责受控 ReAct 与工具使用。

#### C. Hybrid Research Subgraph
负责复杂混合任务的任务拆解与结果聚合。

## 7. RAG 主链路设计

### 7.1 文档接入
1. 文件上传
2. 类型识别
3. 文件去重
4. 原文存储
5. 创建文档记录
6. 发起解析任务

### 7.2 结构化解析
目标是保留：
- 标题层级
- 页码
- 列表
- 表格
- 代码块
- 段落顺序
- section path

### 7.3 Chunk 设计
采用 **Parent-Child 双层 Chunk**：
- Parent Chunk：较大语义块，便于 small-to-big
- Child Chunk：较小检索块，便于精准召回

### 7.4 Contextual Chunking
对每个 child chunk，在向量化前补入：
- 文档标题
- 章节路径
- 上位主题
- 必要上下文摘要

### 7.5 检索策略
采用 **Hybrid Retrieval**：
- Dense Retrieval（向量检索）
- Lexical Retrieval（全文检索 / BM25 风格）
- Metadata Filtering（文档、标签、时间、工作区过滤）

### 7.6 重排序
对合并后的候选结果做 rerank，再选出最终证据块。

### 7.7 证据分级
- sufficient
- weak
- insufficient
- conflicting

### 8.1 Tool Subgraph 的职责
- 规划工具步骤
- 选择工具
- 风险检测
- 审批中断

- 外部副作用调用
- 高风险 MCP 工具

## 9. 记忆设计

系统采用四层记忆：

### 9.1 Working Memory
- 当前线程消息
- 当前检索结果
- 当前工具输出
- 当前中间状态

### 9.2 Episodic Memory
- 过去任务的过程经验
- 失败原因
- 某类问题的处理路径

### 9.3 Semantic Memory
- 用户偏好
- 稳定事实
- 实体映射
- 工作区常识

### 9.4 Procedural Memory
- 某类任务的处理策略
- 工具优先级
- 领域回答规则

---

## 10. Human-in-the-Loop 设计

系统在以下动作前默认进入审批：
- 修改数据库
- 写文件
- 删除资源
- 高风险工具组合调用

执行流程：
1. Graph 运行至风险节点
2. 节点触发 interrupt
3. 前端展示审批卡片
4. 用户选择 approve / reject / modify
5. Graph resume
6. 后续节点继续执行

## 14. 面试表达重点

1. 为什么不是普通 RAG，而是 Agentic RAG
2. 为什么用 LangGraph 做编排
3. 为什么采用 Parent-Child Chunk 与 Contextual Chunking
4. 为什么要做 Hybrid Retrieval + Rerank
5. 为什么记忆要分层
6. 为什么工具调用要可审计、可中断、可恢复
7. 为什么前端必须做过程可视化
8. 为什么 MCP 是未来扩展能力的重要接口层

---

## 15. 结论

本项目最优路线不是“先堆功能”，而是：

**先把高质量 RAG 主链路做扎实，再逐步增加 Agent 编排、Tool/MCP、记忆和前端增强。**
