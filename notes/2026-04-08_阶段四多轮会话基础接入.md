# 阶段四：多轮会话基础接入

## 本轮目标

- 让前端真正复用同一个 `thread_id` 连续发问
- 为 RAG 主链路补一个基础的 follow-up rewrite
- 让多轮追问至少能利用上一问上下文，而不是每一问都当成全新会话

## 后端改动

- `backend/app/services/chat/service.py`
  - graph 初始 state 增加 `rewritten_message`
- `backend/app/services/orchestration/state.py`
  - state 类型补充 `rewritten_message`
- `backend/app/services/orchestration/tracing.py`
  - trace snapshot 增加 `rewritten_message`
- `backend/app/services/orchestration/graph.py`
  - RAG 子链路新增 `rag.rewrite_follow_up`
  - 对短追问做基础 follow-up rewrite
  - `rag.normalize_query` 现在优先使用 `rewritten_message`
  - grounded answer 生成阶段也优先使用 `rewritten_message`
  - 精确问答压缩规则补充 `可以 / 允许`

## 前端改动

- `frontend/src/App.tsx`
  - Chat 页新增当前活动线程状态
  - 线程列表可点选继续会话
  - 新增 `New Thread`
  - 发问时会带上当前 `thread_id`
  - 对话面板展示当前线程最近消息
  - `rag.rewrite_follow_up` 已接入 Live Graph Progress 文案
- `frontend/src/styles/index.css`
  - 补线程上下文展示样式

## 验证

- `pytest backend/tests`：`15 passed`
- `npm run build`：通过
- 真实多轮 smoke：
  - 第 1 问：`Viewer 可以导出引用吗？`
  - 第 2 问：`如果是 Business 呢？`
  - 第 2 问返回：
    - `在 Business 套餐中，Viewer 可以使用问答功能... 与 Starter 不同，Business 下默认允许导出引用内容`
  - `thread_id` 复用成功
  - `GET /api/threads/{thread_id}` 返回消息数：`4`

## 当前边界

- 这一步完成的是“多轮会话基础接入”，不是完整 Memory 系统
- semantic / episodic / procedural memory 仍未实现
- follow-up rewrite 目前是基础版，复杂追问仍可能需要后续继续打磨
