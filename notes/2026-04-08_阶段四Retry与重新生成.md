# 阶段四：Retry 与重新生成

## 本轮目标

- 为已完成或已取消的 run 提供一键重试能力
- 复用现有 LangGraph + SSE + token streaming 链路
- 在前端补 `Retry Last Run / Regenerate / Retry Selected Run`

## 后端改动

- 在 `backend/app/services/chat/service.py` 增加 `rebuild_chat_request(run_id)`，可从历史 `run -> request_message -> thread` 还原新的 `ChatRequest`
- 在 `backend/app/api/routes/runs.py` 新增：
  - `POST /api/runs/{run_id}/retry`
  - `POST /api/runs/{run_id}/retry/stream`
- 在 `backend/app/api/routes/chat.py` 抽出 `build_chat_stream_response(...)`，让普通提问和 retry 共用同一套流式执行器
- SSE 事件补充 `question` 与 `retry_of_run_id`，前端无需重新输入问题也能显示本次“重新生成”的来源
- `prepare_chat(...)` 现在会把 `request_metadata` 一并写入 user message metadata，后续重建请求时可保留入口上下文

## 前端改动

- `frontend/src/App.tsx` 将流式聊天处理抽成一套公共方法
- 聊天区新增：
  - `Retry Last Run`
  - 每条历史回答的 `Regenerate`
- Trace Center 新增：
  - `Retry Selected Run`
- retry 仍然复用现有 `Live Graph Progress`、`Live Answer Draft`、`Trace` 跳转逻辑
