# 阶段四：Step 状态流式化

## 本轮目标

- 将流式 `step` 事件从“只在完成后追加一条摘要”升级为“started/running/completed/cancelled/failed 生命周期”
- 让前端对同一 graph step 原地更新状态，而不是不断堆叠重复行
- 保持 Trace Center、SSE、取消逻辑的语义一致

## 后端改动

- `backend/app/services/orchestration/tracing.py`
  - 为 `RunTraceRecorder` 增加 step lifecycle callback
  - `instrument(...)` 在 step 开始时发 `running`
  - step 正常结束时发 `completed`
  - `RunCancelledError` 显式映射成 `cancelled`
  - 其他异常继续映射为 `failed`
- `backend/app/services/orchestration/graph.py`
  - 增加 `set_step_event_callback(...)`
- `backend/app/api/routes/chat.py`
  - SSE 的 `step` 事件改为走 recorder lifecycle callback
  - graph stream 循环继续只负责合并状态和提交事务，不再重复拼“完成态 step 事件”

## 前端改动

- `frontend/src/App.tsx`
  - `StreamStep` 增加 `stepType`、`status`、`errorMessage`
  - 新增 `upsertStreamStep(...)`，按 `stepKey` 原地更新
  - `Live Graph Progress` 现在显示：
    - step 名称
    - step 类型
    - 状态 pill
    - 运行摘要或错误信息
- `frontend/src/styles/index.css`
  - 增加 `running / completed / cancelled / failed` 的状态视觉区分
  - 移动端下对 step meta 做了左对齐修正

## 验证

- `pytest backend/tests`：`13 passed`
- `npm run build`：通过
- 真实 success smoke：
  - `POST /api/runs/{run_id}/retry/stream` 返回 `200`
  - step 样本序列包含：
    - `load_thread_context: running -> completed`
    - `classify_request: running -> completed`
    - `invoke_rag_subgraph: running -> completed`
- 真实 cancel smoke：
  - 临时 `uvicorn` 进程 + 独立 HTTP 客户端发起 stream
  - `POST /api/runs/{new_run_id}/cancel` 返回 `accepted = true`
  - SSE 收到 `cancelled`
  - `HAS_CANCELLED_STEP = True`
