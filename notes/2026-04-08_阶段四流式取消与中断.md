# 2026-04-08 阶段四补充：流式取消与中断

## 本轮目标

- 为流式聊天增加真正的取消能力
- 保持 token streaming、step streaming、Trace 落库和同步聊天能力不回归
- 让前端可以停止当前 run，而不是只能等待生成完成

## 已完成内容

### 1. 流式取消注册表

- 新增 `StreamCancellationRegistry`
- 新增 `RunCancelledError`
- 当前活跃 stream run 会注册一个取消信号
- 取消后会由图编排和模型流式层共同响应

关键文件：

- `backend/app/services/chat/stream_control.py`

### 2. 图编排支持中断

- `AtlasSupervisorGraphService` 新增：
  - `set_cancel_checker(...)`
  - `_raise_if_cancelled()`
- 关键节点都会在执行前检查取消状态
- `direct` 和 `rag.generate_grounded_answer` 的流式生成阶段也会检查取消状态

关键文件：

- `backend/app/services/orchestration/graph.py`

### 3. DashScope 流式客户端支持取消

- `DashScopeClient.chat_stream(...)` 新增 `cancel_checker`
- 当取消信号被置位时，会主动中断当前流式读取并抛出 `RunCancelledError`

关键文件：

- `backend/app/services/llm/dashscope_client.py`

### 4. 后端流式路由支持 cancelled 事件

- `POST /api/chat/stream` 现在由后台线程执行
- 主线程通过事件队列持续返回 SSE
- 新增 `cancelled` 事件
- run 被取消后会落库为：
  - `status = cancelled`
  - `result_status = cancelled`
  - `error_code = RunCancelledError`

关键文件：

- `backend/app/api/routes/chat.py`
- `backend/app/services/chat/service.py`

### 5. 新增取消接口

- 新增 `POST /api/runs/{run_id}/cancel`
- 前端拿到 `run_id` 后可请求服务端取消当前 stream run

关键文件：

- `backend/app/api/routes/runs.py`

### 6. 前端 Stop 按钮

- `Chat Workspace` 在流式生成期间显示 `Stop`
- 点击后调用 `POST /api/runs/{run_id}/cancel`
- 前端会显示：
  - `Stopping generation...`
  - 收到 `cancelled` 后保留当前草稿
  - run 仍可在 `Trace Center` 中查看

关键文件：

- `frontend/src/App.tsx`
- `frontend/src/styles/index.css`

### 7. 新增测试

- 新增取消注册表回归测试
- 当前编排测试总数提升到 `10`

关键文件：

- `backend/tests/test_orchestration_trace.py`

### 真实取消 smoke test

测试方式：

- 启动临时本地 uvicorn
- 通过真实 HTTP 调用 `POST /api/chat/stream`
- 在收到 `run_created` 后立刻调用 `POST /api/runs/{run_id}/cancel`

结果：

- Stream HTTP 状态：`200`
- 取消接口返回：
  - `accepted = true`
  - `status = cancelling`
- SSE 最终收到：
  - `cancelled`
- Trace 查询结果：
  - `run.status = cancelled`
  - `run.error_code = RunCancelledError`

## 当前边界

- 取消是 cooperative cancellation，不是强制杀线程
- 如果取消发生在外部 API 阻塞读取的极短窗口内，可能会在下一个检查点才生效
- 当前取消后不会把部分答案持久化为 assistant message，只保留前端草稿和 run trace

## 下一步建议

1. 增加前端“重新生成”能力
2. 为取消后的 run 增加更清晰的前端状态标识
3. 再往后进入 Memory 节点和多轮状态编排
